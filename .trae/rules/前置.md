# ‌本项目基于**团结引擎 Unity 2022.3.62f1**‌ ，下面为性能优化最佳实践，主要关注在游戏开发中如何高效利用 Unity 引擎的功能，避免在热路径中出现分配操作。由于是大型网络游戏项目，需要代码质量、可维护性和可扩展性并能有极致的性能表现。

## 一、核心铁律（优先级最高）

1. **热路径零分配**：`Update`、`FixedUpdate`、`OnEnable`、循环体内**禁止**出现任何引用类型（class）的 `new` 操作。必须使用 `UnityEngine.Pool.ObjectPool`（**仅主线程安全**）、`System.Buffers.ArrayPool<T>` 或预分配缓存。
2. **极致缓存机制**：`GetComponent`、`Find`、`Resources.Load`、`Camera.main` 严禁出现在热路径中。所有组件引用、`Shader.PropertyToID` 哈希值必须在 `Awake` 或 `OnEnable` 中完成缓存。
3. **优先值类型（Struct）**：纯数据载体（**建议 ≤ 16 字节**）强制使用 `struct` 定义；方法参数若非必要，禁止使用 `ref` 或 `out` 传递大对象，避免不必要的防御性拷贝（若超过 16 字节，可考虑 `ref` 传递以提升性能）。
4. **物理查询禁止分配**：禁止使用 `RaycastAll`、`OverlapSphereAll` 等返回数组的 API，必须使用 `NonAlloc` 系列方法（如 `RaycastNonAlloc`）配合预定义的静态数组。
5. **标签比较零分配**：禁止使用 `gameObject.tag == "xxx"`（会产生字符串分配），必须使用 `gameObject.CompareTag("xxx")`。
6. **协程等待缓存**：禁止在协程中每次 `new WaitForSeconds()` 或 `new WaitForEndOfFrame()`，必须使用静态缓存的实例（例如 `static readonly WaitForSeconds s_wait = new WaitForSeconds(0.5f);`）。**若等待时间需要动态变化，则使用 `new` 是合理的，不可盲目缓存固定值。**

## 二、引擎交互与 API 规范

1. **字符串与哈希**：`Animator`、`Material`、`Shader` 的属性传递必须使用 `Shader.PropertyToID` 获取 int 哈希，严禁在循环中使用字符串。
2. **距离与比较**：判断距离远近必须使用 `sqrMagnitude`（平方距离），严禁直接使用 `Vector3.Distance`（避免开方运算）。
3. **循环迭代**：集合遍历应优先使用 `for` 循环。对于 `List<T>`，现代 Unity 版本（IL2CPP）下 `foreach` 不会产生 GC 分配，但为了兼容性和明确性，推荐在热路径中使用 `for`；`NativeArray`/`NativeList` 的枚举器无分配，可放心使用。
4. **日志隔离**：所有 `Debug.Log` 必须使用 `#if UNITY_EDITOR` 宏包裹，或在构建脚本中定义 `DISABLE_LOG` 宏，严禁在生产包中输出运行日志。推荐使用 `[Conditional("UNITY_EDITOR")]` 自定义日志方法。
5. **Camera.main 缓存**：`Camera.main` 内部会调用 `Find`，必须在初始化阶段缓存，热路径使用缓存变量。
6. **材质属性修改**：禁止直接修改 `material.color` 等属性（会导致材质实例化），必须使用 `MaterialPropertyBlock` 并调用 `renderer.SetPropertyBlock(mpb)`。
7. **Draw Call 优化**：大量静态环境物件可调用 `StaticBatchingUtility.Combine` 进行运行时静态合批；大量动态物件优先使用 `Graphics.DrawMeshInstanced` 或 `DrawMeshInstancedIndirect` 实现 GPU Instancing。
8. **Shader 预热**：使用 `ShaderVariantCollection` 收集变体，在加载界面调用 `WarmUp()` 预热，消除首次渲染卡顿。构建时启用 **Shader Stripping**（剔除无用变体）以减小包体并缩短预热时间。
9. **GPU 回读**：需要从 GPU 读取数据时，必须使用 `AsyncGPUReadback.Request` 进行异步回读，严禁阻塞主线程。**该 API 在 DX12、Vulkan、Metal 等现代图形 API 上受支持，使用前请确认目标平台。**
10. **异步场景加载**：必须使用 `SceneManager.LoadSceneAsync` 并允许通过 `allowSceneActivation` 控制激活时机，严禁使用已过时的 `Application.LoadLevel`。
11. **Transform 局部缓存**：在循环或频繁调用的函数中，若需多次读写 `Transform.position`、`rotation` 等，应先将 `Transform` 引用缓存到局部变量，减少跨域调用开销。

## 三、架构灵活性与设计模式

1. **解耦与扩展**：模块间依赖基于接口（Interface）编程；配置数据（数值、预制体引用）必须使用 `ScriptableObject`，禁止硬编码。
2. **事件系统**：使用 `UnityEvent` 或弱引用委托时，必须在 `OnDisable` 中主动注销，避免内存泄漏与无效调用。
3. **异步加载**：资源加载优先使用 `Addressables` 并配合 `using` 语句管理引用计数，严禁同步阻塞主线程。场景加载可预先 `LoadSceneAsync` 并延迟激活。
4. **音频池化**：禁止频繁创建和销毁 `AudioSource` 组件或挂载音频的 GameObject（`AudioSource.PlayOneShot` 本身不会分配新 AudioSource，问题在于对象本身的创建/销毁）。应预创建 `AudioSource` 对象池并循环使用。
5. **单例与 Manager**：若涉及 Manager 或单例，提供简洁的调用示例（代码中以注释形式 `// Example: ...`）。

## 四、计算、数据结构与并行处理

1. **数学库**：复杂数学计算（矩阵、大批量向量）优先使用 `Unity.Mathematics` 中的 `float3`、`float4x4`，并配合 `[BurstCompile]` 标签。
2. **集合容量预分配**：频繁增删的 `List<T>` 初始化时预设容量 `new List<T>(capacity)`；字典操作优先使用 `TryGetValue` 避免双重索引。
3. **多线程与 Job System**：主线程密集计算（寻路、粒子运动等）必须拆分为 `IJobParallelFor` 作业，并全部使用 `NativeArray`、`NativeHashMap` 等 `NativeContainer` 传递数据。
4. **Burst 编译**：任何可独立于主线程的计算作业结构体必须标记 `[BurstCompile]`，以获得极致性能。
5. **字符串处理**：热路径中避免字符串拼接，使用 `StringBuilder`（预缓存并 `Clear()` 复用）或 `Unity.Collections.FixedString` 进行 Burst 友好的操作。
6. **临时数组**：需要临时数组时，从 `ArrayPool<T>.Shared` 租用，用完归还，避免 GC。

## 五、渲染、UI 与资源管理优化

1. **Sprite 渲染**：大量动态 Sprite 应使用 `SpriteAtlas` 打包，启用 Sprite Atlas Manager 减少 Draw Call。**注意：在 URP/HDRP 管线中，SRP Batcher 可能部分取代 Atlas 的作用，请根据项目渲染管线选择最佳方案。**
2. **UI 画布拆分**：将频繁变化的 UI 元素（血条、飘字）放在独立 Canvas 中，避免污染静态画布导致整体网格重建。
3. **UI 射线隔离**：不可交互的 Text、Image 必须关闭 `RaycastTarget`，减少 `GraphicRaycaster` 排序与检测开销。
4. **文本组件**：任何需要动态更新文字的场景必须使用 `TextMeshPro`，严禁使用默认 `Text`。
5. **UI 布局**：避免 `Layout Group` 深度嵌套，尽可能手动布局或写死尺寸，防止多次布局重建。频繁修改 UI 颜色时，可考虑直接设置 `CanvasRenderer.SetColor` 跳过图形重建。
6. **资源卸载**：场景切换后可一次性调用 `Resources.UnloadUnusedAssets` 清理残留，**但该方法会阻塞主线程，耗时极高**，禁止频繁调用。更推荐使用 Addressables 的细粒度资源管理，按需加载/卸载。
7. **Canvas 启用方式**：需要隐藏/显示 UI 时，优先使用 `canvas.enabled = false` 而非 `gameObject.SetActive(false)`，因为后者会触发层级结构重建，开销更大。
8. **移除空 Update**：如果脚本中包含 `Update()`、`FixedUpdate()` 等方法但方法体为空，请将其删除。空方法仍会被 Unity 的消息系统调用，产生不必要的开销。

## 六、网络与序列化

1. **序列化**：网络同步或频繁序列化场景，禁止使用 `BinaryFormatter` 或带大量字符串分配的方案，优先使用 `Netcode` 内置序列化（如 `FastBufferWriter`）或 `FixedString`。
2. **缓冲区管理**：网络消息缓冲区从 `ArrayPool<byte>.Shared` 租用，发送完成即归还，避免频繁 `new byte[]`。

## 七、负面清单（绝对禁止 / 谨慎使用）

- 禁止在运行时使用 `GameObject.Find` / `FindObjectOfType`（编辑器工具除外）。
- 禁止在 `Update` 中调用 `Camera.main`（必须缓存）。
- 禁止在未加 `[RequireComponent]` 时，在 `Awake` 中假设其他组件必定存在。
- 禁止在循环体内直接访问 `Transform.position` / `Transform.rotation`（必须缓存到本地变量再进行数学运算）。
- 禁止在频繁调用的函数中创建非静态的 `Action` / `Func` 匿名委托（除非使用 `static` 修饰且无捕获变量）。
- 禁止使用 `gameObject.tag == "xxx"` 进行标签判断（必须用 `CompareTag`）。
- 禁止直接修改 `material.color`（必须用 `MaterialPropertyBlock`）。
- 禁止在协程中每帧 `new WaitForSeconds()`（必须静态缓存，除非动态时间）。
- **谨慎使用** `foreach` 遍历 `List<T>` 等集合：现代 Unity（IL2CPP）下无 GC 分配，但若项目使用 Mono 后端或旧版本，仍可能产生分配，建议在热路径中优先使用 `for` 并根据测试结果决定。
- **优先使用异步加载**：大型资源必须使用 `Addressables.LoadAssetAsync` 或 `Resources.LoadAsync`，严禁同步 `Resources.Load` 阻塞主线程；极小资源（如配置文件）可酌情同步加载，但需评估影响。
- **禁止在热路径中对 `UnityEngine.Object` 进行 `null` 比较**（如 `if (gameObject == null)`），因为其重载了 `==` 运算符，开销远大于普通 `null` 检查。应使用 `ReferenceEquals` 或缓存有效状态，或在非热路径中谨慎使用。
- **禁止使用 `Application.LoadLevel`**（已过时），必须使用 `SceneManager.LoadScene` 系列。

## 八、输出格式与质量要求

1. 直接输出完整的 `.cs` 文件源码，包含必要的 `using` 和命名空间声明。
2. 在关键优化点（如对象池拉取、`NonAlloc` 调用、缓存字典、Job 调度）上方添加 `// GC Optimized` 或 `// Performance Critical` 注释。
3. 若涉及 Manager 或单例，提供简洁的调用示例（`// Example: ...`）。
4. 拒绝生成过时或冗余的 API（如 `Application.LoadLevel`、`UnityEngine.RuntimePlatform` 的某些旧枚举等），仅提供当前 Unity 主流推荐方案。

## 九、Playable 与动画系统优化

1. **Graph 即资源**：每个 `Animator` 只应绑定一个 `PlayableGraph`，在 `Awake/OnEnable` 中创建并播放一次，在 `OnDisable/OnDestroy` 中销毁。严禁在热路径中创建、播放或销毁 Graph。
2. **节点复用**：动态切换动画使用固定 `AnimationMixerPlayable`，通过 `Connect`/`Disconnect` 或权重切换；禁止每切换一次就 `Create/Destroy` `AnimationClipPlayable`。
3. **零分配回调**：`ScriptPlayable` 的 `PrepareFrame` / `OnBehaviourPause` 必须零分配；推荐使用 `IAnimationJob` + `AnimationScriptPlayable` 搭配 Burst。
4. **稳定拓扑**：预分配 `AnimationMixerPlayable` 的输入端口数量，避免运行时扩展导致内部分配。
5. **避免便捷方法**：禁止在运行时使用 `AnimationPlayableUtilities.PlayClip` 等“一步式”便捷 API。
